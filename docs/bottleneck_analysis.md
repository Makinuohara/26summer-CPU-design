访存瓶颈分析报告


1. 测试环境

测试平台：Vivado 2018.3行为级仿真
被测模块：cache.v，直接映射Cache，8行乘1字
统计信号：hit_count和miss_count，均为32位寄存器
测试程序：cache_tb.v，执行4组典型访存操作


2. 仿真结果

2.1 控制台输出摘要

性能统计结果：
命中次数：2
未命中次数：2
总访问次数：4
命中率：百分之五十
测试结论：Cache命中功能正常

2.2 各测试用例详细分析

测试用例1：写地址0x04，数据0x12345678
期望结果：Miss
实际结果：Cache未命中，写入Cache并置有效位
命中计数：0
未命中计数：1

测试用例2：读同一地址0x04
期望结果：Hit
实际结果：命中，读出0x12345678
命中计数：1
未命中计数：1

测试用例3：读不同地址0x10
期望结果：Miss
实际结果：未命中，Cache填充0xAAAAAAAA
命中计数：1
未命中计数：2

测试用例4：再次读地址0x10
期望结果：Hit
实际结果：命中，读出0xAAAAAAAA
命中计数：2
未命中计数：2

2.3 波形验证

在Vivado波形窗口中观察到以下行为：
测试1中写操作触发write_en为高，miss信号在下一个时钟沿拉高。
测试2中read_en为高时，hit信号拉高，read_data正确输出0x12345678。
测试3中地址切换后hit_internal为低，触发Miss，cache_data更新为填充值0xAAAAAAAA。
测试4中同一地址再次访问，hit_internal为高，直接返回0xAAAAAAAA。


3. 瓶颈分析

3.1 当前性能瓶颈

整体命中率仅为百分之五十，即4次访问中命中2次、未命中2次。主要原因如下：

冷启动缺失：测试1和测试3均为首次访问该地址，Cache初始为空，必然产生缺失。这是影响当前命中率的主要因素。

Cache容量小：仅8行存储空间，若测试程序规模增大，容量缺失将快速上升。

直接映射冲突：多个地址映射到同一Index时相互替换，造成冲突缺失。当前测试序列中地址0x04和0x10的Index分别为1和4，未发生冲突，但后续扩展测试时需重点关注。

3.2 深层分析

测试序列中的缺失全部来自首次访问，即冷启动缺失。地址0x04的Index为1，地址0x10的Index为4，两者Index不同，不存在冲突缺失。

若测试程序增加访问跨度更大或更密集的地址序列，冲突缺失将成为主要瓶颈。

当前Cache读Miss时填充固定值32位十六进制AAAAAAAA，缺乏与主存（dmem）的真实交互，未体现实际访存延迟。在实际系统中，主存访问延迟如2到3个周期会直接增加CPU停顿周期，进一步加剧性能损失。


4. 改进建议

4.1 短期优化（当前架构内）

接入真实主存：Cache读Miss时从dmem或imem读取数据，而非填充固定值，可提升数据准确性。

增加Cache行数：从8行增至16行或32行，预期命中率可提升百分之十到二十。

增加预取机制：对顺序访问模式提前加载下一行数据，使顺序程序命中率接近百分之百。

4.2 长期优化（架构升级）

提高相联度：改为两路组相联，允许更多地址映射到同一组，显著降低冲突缺失。

优化替换策略：从直接映射改为LRU或随机替换，提升随机访问的命中率。

增加Cache层级：引入L2 Cache，降低主存访问压力。


5. 结论

当前Cache模块功能正确，命中率统计机制工作正常。实测命中率为百分之五十，主要受冷启动缺失影响。改进方向明确。在系统集成时，建议优先将Cache读Miss处理接入真实主存，并适当增大Cache容量，以提升整体系统性能。
