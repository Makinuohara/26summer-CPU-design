# 基于 FPGA 开发板的处理器设计分层需求文档

## 1. 文档目的

本文档依据课程 PPT 中“题目 B：基于 FPGA 开发板的处理器设计”的三层次任务要求，整理本课设的需求边界、功能目标、验证方式与交付物。文档用于指导后续设计、实现、测试、上板验证和课程报告撰写。

## 2. 项目背景

本项目面向 FPGA 开发板实现一个处理器系统。设计以 RISC-V RV32I 子集为主要参照，也可兼容课程要求中提到的 MIPS 或自定义 ISA 思路。当前工程选择 RISC-V RV32I 子集，使用 Verilog 实现 CPU 数据通路、控制器、存储器和板级调试显示逻辑，并在 Nexys 4 DDR 平台完成综合、布局布线和硬件验证。

## 3. 总体需求

### 3.1 设计目标

项目应按“基础层次、进阶层次、拓展层次”逐步推进：

| 层次 | 名称 | 核心目标 |
| --- | --- | --- |
| 基础层次 | 简单 CPU 设计与实现 | 实现支持基本指令集的单周期或多周期 CPU，并完成仿真与上板验证 |
| 进阶层次 | 完整计算机系统设计 | 在 CPU 基础上集成内存与 I/O 接口，形成可运行小型程序的系统 |
| 拓展层次 | 高性能处理器优化设计 | 在完整系统基础上进行性能优化或创新探索，并给出量化分析 |

### 3.2 目标平台

| 项目 | 要求 |
| --- | --- |
| 硬件平台 | Nexys 4 DDR FPGA 开发板 |
| HDL 语言 | Verilog |
| 综合实现工具 | Vivado |
| 仿真工具 | Icarus Verilog 或 Vivado Simulator |
| 观察接口 | LED、拨码开关、七段数码管，后续可扩展 UART 或更多 I/O |

### 3.3 设计约束

1. CPU 指令集应有明确说明，包括指令格式、操作语义和支持范围。
2. 数据通路和控制器应模块化实现，便于扩展指令和替换 CPU 结构。
3. 所有层次均应提供可复现实验步骤，包括仿真命令、综合脚本和上板观察方法。
4. 验证结果应能说明指令执行、访存、跳转和 I/O 观察链路正确。
5. 拓展层次的优化应给出指标对比，至少覆盖频率、CPI、吞吐量、资源或功耗中的若干项。

## 4. 基础层次需求：简单 CPU 设计与实现

### 4.1 层次目标

设计并实现一个支持基本指令集的单周期或多周期 CPU。CPU 应以 RISC-V RV32I 子集为参照，完成指令集、数据通路和控制器设计，能够运行简单测试程序，并在 FPGA 开发板上完成硬件验证。

当前工程采用单周期 CPU 方案。

### 4.2 功能需求

| 编号 | 需求 | 说明 | 当前状态 |
| --- | --- | --- | --- |
| B-F-01 | 实现 PC 更新逻辑 | 支持顺序执行、条件分支和跳转目标选择 | 已完成 |
| B-F-02 | 实现指令存储器 | 能根据 PC 输出当前指令 | 已完成 |
| B-F-03 | 实现寄存器堆 | 32 个 32 位通用寄存器，`x0` 恒为 0 | 已完成 |
| B-F-04 | 实现立即数生成 | 支持 I/S/B/J 型立即数 | 已完成 |
| B-F-05 | 实现 ALU | 支持加、减、与、或、异或等基本运算 | 已完成 |
| B-F-06 | 实现主控制器 | 根据 opcode 生成写回、访存、分支、跳转等控制信号 | 已完成 |
| B-F-07 | 实现 ALU 控制器 | 根据 `alu_op`、`funct3`、`funct7` 选择 ALU 操作 | 已完成 |
| B-F-08 | 实现数据存储器 | 支持 32 位字读写 | 已完成 |
| B-F-09 | 实现分支判断 | 至少支持 `beq`、`bne` | 已完成 |
| B-F-10 | 实现板级观察逻辑 | 通过 LED、拨码开关或数码管观察关键结果 | 已完成 |

### 4.3 指令需求

基础层次至少应支持算术、逻辑、访存、跳转等基本指令类型。

| 类型 | 指令 | 当前状态 |
| --- | --- | --- |
| 算术 | `add`、`sub`、`addi` | 已完成 |
| 逻辑 | `and`、`or`、`xor`、`andi`、`ori`、`xori` | 已完成 |
| 访存 | `lw`、`sw` | 已完成 |
| 分支 | `beq`、`bne` | 已完成 |
| 跳转 | `jal` | 已完成 |

### 4.4 验证需求

| 编号 | 验证项 | 验证方法 | 通过标准 |
| --- | --- | --- | --- |
| B-V-01 | 指令取指与 PC 更新 | 仿真观察 PC 和指令序列 | PC 按 `PC+4`、分支目标或跳转目标更新 |
| B-V-02 | 算术运算 | 运行加法、减法测试程序 | 寄存器写回结果正确 |
| B-V-03 | 逻辑运算 | 运行与、或、异或测试程序 | 寄存器写回结果正确 |
| B-V-04 | 访存链路 | 执行 `sw` 后再执行 `lw` | 读回数据与写入数据一致 |
| B-V-05 | 分支跳转 | 执行 `beq`、`bne`、`jal` | 程序进入预期路径 |
| B-V-06 | 上板验证 | 下载 bitstream 后观察 LED/数码管 | 能观察到成功标志和访存结果 |

当前 smoke test 的通过标准为：

```text
x5 = 1
data_mem[0] = 0x0000000c
```

## 5. 进阶层次需求：完整计算机系统设计

### 5.1 层次目标

在基础 CPU 上集成内存与 I/O 接口，构建能够完整运行小型测试程序的计算机系统。该层次不仅关注 CPU 内核本身，还要求 CPU、内存子系统和基本 I/O 之间形成可运行、可观察、可分析的整体。

### 5.2 功能需求

| 编号 | 需求 | 说明 | 当前状态 |
| --- | --- | --- | --- |
| A-F-01 | 完善指令集覆盖 | 扩展 RV32I 常用指令，如移位、比较、`lui`、`auipc`、`jalr` | 待完成 |
| A-F-02 | 完善分支支持 | 支持 `blt`、`bge`、`bltu`、`bgeu` | 待完成 |
| A-F-03 | 规划内存子系统 | 明确指令存储器、数据存储器、地址空间和访问规则 | 待完成 |
| A-F-04 | 集成基本 I/O 接口 | 支持按键、拨码开关、LED、数码管，或 UART 等外设 | 部分完成 |
| A-F-05 | 形成系统级顶层 | 将 CPU、内存和 I/O 统一接入一个可综合顶层 | 部分完成 |
| A-F-06 | 支持小型测试程序 | 程序应覆盖循环、数组访问、分支和 I/O 输出 | 待完成 |
| A-F-07 | 支持系统状态观察 | 能选择观察寄存器、内存地址、PC 或外设状态 | 待完成 |

### 5.3 内存与 I/O 需求

| 模块 | 要求 |
| --- | --- |
| 指令存储器 | 支持加载更长测试程序，后续可由 hex 文件初始化 |
| 数据存储器 | 支持字访问，后续可扩展字节和半字访问 |
| I/O 映射 | 建议采用 memory-mapped I/O，便于 CPU 通过 `lw/sw` 访问外设 |
| 显示输出 | 至少支持 LED 或七段数码管显示程序结果 |
| 输入控制 | 至少支持拨码开关或按键输入，用于选择显示内容或控制程序 |

### 5.4 性能分析需求

课程 PPT 要求进阶层次能够分析系统性能瓶颈并提出优化方案，同时引入流水线机制，对时钟频率、CPI 与吞吐量进行量化评估。因此进阶层次应至少形成以下分析结果：

| 指标 | 说明 |
| --- | --- |
| 时钟频率 | Vivado 综合或实现后的最大工作频率 |
| CPI | 单周期 CPU 固定约为 1，多周期或流水线需按程序统计 |
| 吞吐量 | 单位时间完成指令数，可由频率和 CPI 估算 |
| 资源占用 | LUT、FF、BRAM 等 FPGA 资源 |
| 瓶颈分析 | 说明关键路径或访存/I/O 对性能的影响 |

| 编号 | 验证项 | 通过标准 |
| --- | --- | --- |
| A-V-01 | 扩展指令仿真 | 每类扩展指令都有对应测试并通过 |
| A-V-02 | 系统程序运行 | 小型测试程序能完整运行到预期结束状态 |
| A-V-03 | I/O 访问 | CPU 能通过统一接口读取输入或写出显示结果 |
| A-V-04 | 上板演示 | 开发板上可观察程序运行结果或状态变化 |
| A-V-05 | 性能报告 | 给出频率、CPI、吞吐量和资源占用分析 |

## 6. 拓展层次需求：高性能处理器优化设计

### 6.1 层次目标

在完整计算机系统基础上进行多维度性能优化与创新探索。该层次要求从给定方向中选择至少一项深入实现，并通过实验数据说明设计权衡。

### 6.2 可选优化方向

| 编号 | 方向 | 说明 | 当前状态 |
| --- | --- | --- | --- |
| E-F-01 | 流水线冒险完整解决方案 | 实现数据前递、暂停、分支冲刷或分支预测 | 待完成 |
| E-F-02 | Cache 替换策略优化 | 设计 Cache，并比较不同替换策略与命中率 | 待完成 |
| E-F-03 | 浮点或乘除法扩展 | 支持浮点运算或整数乘除法扩展指令 | 待完成 |
| E-F-04 | RISC-V 自定义 ISA 扩展 | 基于 RISC-V 扩展自定义指令 | 待完成 |
| E-F-05 | AI/矩阵计算加速 | 设计 MAC 乘加单元或矩阵计算加速指令 | 待完成 |
| E-F-06 | 多方案 PPA 对比 | 从功耗、性能、面积角度分析设计权衡 | 待完成 |

### 6.3 推荐实现路线

结合当前工程基础，推荐优先选择“流水线 CPU 与冒险处理”作为拓展层次主线。原因是当前单周期 CPU 已经具备清晰的数据通路和控制信号，适合拆分为 IF、ID、EX、MEM、WB 五级流水结构。

推荐路线如下：

1. 将当前单周期数据通路拆分为五级流水。
2. 添加 IF/ID、ID/EX、EX/MEM、MEM/WB 流水寄存器。
3. 处理 load-use 数据冒险，必要时插入暂停。
4. 实现 ALU 结果前递，减少不必要暂停。
5. 对分支和跳转进行冲刷处理。
6. 使用同一测试程序对比单周期和流水线的 CPI、频率、吞吐量与资源占用。

### 6.4 验证需求

| 编号 | 验证项 | 通过标准 |
| --- | --- | --- |
| E-V-01 | 功能正确性 | 优化后 CPU 运行测试程序结果与基础 CPU 一致 |
| E-V-02 | 冒险处理 | 数据相关、load-use、分支跳转场景均能正确执行 |
| E-V-03 | 性能对比 | 给出优化前后频率、CPI、吞吐量或资源对比 |
| E-V-04 | PPA 分析 | 说明性能收益、资源代价和设计取舍 |
| E-V-05 | 上板验证 | 优化版本能够综合并在开发板上运行或展示结果 |

## 8. 阶段验收标准

### 8.1 基础层次验收

基础层次通过应满足：

1. RTL 源码可综合，无阻塞性语法错误。
2. 仿真程序运行结束后能得到预期寄存器和内存结果。
3. Vivado 能完成综合、实现并生成 bitstream。
4. Nexys 4 DDR 上能通过 LED 或数码管观察到程序运行结果。
5. 报告中能说明 ISA、数据通路、控制器和验证方法。

### 8.2 进阶层次验收

进阶层次通过应满足：

1. CPU 支持更多 RV32I 常用指令。
2. 系统集成 CPU、内存和至少一种 I/O 接口。
3. 小型测试程序能完整运行，并通过外设输出结果。
4. 能给出系统频率、CPI、吞吐量和资源占用分析。
5. 能指出当前系统性能瓶颈并提出优化方案。

### 8.3 拓展层次验收

拓展层次通过应满足：

1. 至少完成一种性能优化或创新扩展方向。
2. 优化前后均能通过功能测试。
3. 对性能提升和资源代价进行量化比较。
4. 从 PPA 角度说明设计取舍。
5. 能形成可展示、可复现的实验结果。

## 9. 后续实施建议

后续推进建议按以下顺序执行：

1. 补充基础层次测试，覆盖已实现但 smoke test 未单独展示的逻辑指令和 `bne`。
2. 扩展 RV32I 常用指令，优先实现 `sll`、`srl`、`sra`、`slt`、`slti`、`lui`、`auipc`、`jalr`。
3. 将指令 ROM 改为支持 hex 初始化，便于替换测试程序。
4. 设计 memory-mapped I/O，统一 CPU 对 LED、数码管、开关的访问方式。
5. 在系统稳定后选择流水线作为拓展方向，并完成冒险处理和性能对比。
