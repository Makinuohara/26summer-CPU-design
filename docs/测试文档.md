# 测试文档

## 1. 测试目标

本轮测试面向阶段二 SoC 集成版本，验证以下四类功能已经贯通：

- CPU 基本取指、执行与流水线控制
- 存储系统读写与数据通路连通性
- MMIO 外设访问链路
- 外部中断从外设到 CPU trap 再返回主程序的完整流程

## 2. 测试环境

- 开发板：Nexys 4 DDR
- FPGA 器件：`xc7a100tcsg324-1`
- 工具：Vivado 2018.3
- 顶层工程：`fpga_top` / 板级联调版本
- 板级测试程序：[`src/fpga_demo_imem.v`](D:/project/RISC-V-cpu/26summer-CPU-design/src/fpga_demo_imem.v)
- SoC 联调仿真：[`sim/tb_pipeline_irq_demo.v`](D:/project/RISC-V-cpu/26summer-CPU-design/sim/tb_pipeline_irq_demo.v)

## 3. 已完成测试项

### 3.1 MMIO 基础通路测试

测试目的：
验证 CPU 能够通过数据总线访问 `SW`、`LED`、`SEG` 等 MMIO 外设，确认地址译码、写回路径和显示链路正确。

测试方法：

- 在仿真中运行 `lw SW -> sw LED -> sw SEG` 的镜像程序
- 观察 `dmem_bus_decoder` 片选是否正确命中外设
- 观察 LED 与七段数码管是否跟随 CPU 写入变化

测试结论：

- MMIO 地址译码正常
- CPU 对外设的读写链路已打通
- LED 与数码管可作为后续中断联调的板级观测窗口

### 3.2 任务 2：CPU trap 流程测试

测试目的：
验证 CPU 对外部中断的基本响应流程是否正确，包括：

- `meip` 进入 CPU
- CSR 检测 `mie.MEIE`、`mstatus.MIE`
- 保存 `mepc`
- 写入 `mcause = 0x8000_000b`
- `PC` 跳转到 `mtvec`
- 执行中断服务程序
- `mret` 返回主程序

测试方法：

- 主程序循环显示固定值
- 触发一次开关中断
- 中断服务程序将 `LED[4]` 对应标志位置位，并让数码管短暂显示 `EE`
- 返回主程序后继续执行主循环

验证现象：

- 触发中断后，LED 出现中断标记
- 数码管能进入中断标志显示，再回到主程序显示

测试结论：

- CPU trap 入口逻辑正确
- `mtvec/mepc/mcause/mstatus/mie` 相关 CSR 控制链路已实际跑通

### 3.3 任务 3：中断期间 MMIO 访问测试

测试目的：
验证中断服务程序内部执行 MMIO 访问时，既不会破坏主程序状态，也不会导致总线冲突。

测试方法：

- 主程序持续读取 `SW` 并刷新显示
- 触发中断后，在 handler 中再次读取 `SW`
- handler 中写 `LED`、写数码管，随后 `mret` 返回

验证现象：

- 中断期间数码管可短暂显示中断标记
- 返回主程序后，显示恢复为主循环内容
- LED 能反映 handler 内部的写入结果

测试结论：

- 中断期间 MMIO 读写功能正常
- CPU trap 流程与外设总线访问可以同时工作

### 3.4 任务 4：存储系统 + 中断 + MMIO 联调测试

测试目的：
验证普通数据存储访问、MMIO、外部中断三者同时存在时，系统仍能正确运行。

当前板级 demo 的逻辑如下：

- 上电初始化：
  - `dmem[0] = 1`
  - `dmem[1] = 2`
  - `mtvec = 0x100`
  - 打开外部中断使能
- 主循环：
  - 读取 `dmem[0]` 与 `dmem[1]`
  - 相加后写回 `dmem[2]`
  - 同时把结果写到七段数码管
- 中断处理：
  - 读取 `SW`
  - 将 `SW | 0x10` 写到 LED
  - 通过中断控制器 `claim/complete`
  - 把 `dmem[0]` 改写为 `SW | 0x10`
  - 返回主程序

板级预期现象：

- 上电后：`LED = 00`，`SEG = 03`
- 触发 `SW=01` 中断后：`LED = 11`，`SEG = 13`

含义说明：

- `SEG = 03` 表示主程序正在执行 `1 + 2`
- 中断后 handler 把 `dmem[0]` 改成 `0x11`
- 主程序恢复后重新计算，得到 `0x11 + 0x02 = 0x13`

仿真结果：

```text
PASS: irq task-4 demo passed led=0011 seg=13
```

板级结果：

- 已成功复现与仿真一致的显示现象

测试结论：

- CPU、存储、MMIO、外部中断已完成一次完整 SoC 级联调
- 当前版本已经不是单模块冒烟，而是系统级行为验证

### 3.5 中断连续触发测试

测试目的：
验证同一中断源连续多次触发时，系统不会出现“只能响应一次”或“claim 后卡死”的问题，重点检查：

- `io_switches` 的 pending 置位与读清零
- `interrupt_controller` 的 `pending/claimed/complete`
- CPU 多次进入 trap 后仍能正确 `mret` 返回
- 主程序在每次中断后都能继续执行

测试文件：

- [`sim/tb_pipeline_irq_retrigger.v`](D:/project/RISC-V-cpu/26summer-CPU-design/sim/tb_pipeline_irq_retrigger.v)

测试方法：

- 保持当前任务 4 的主程序与 handler 不变
- 依次将开关输入改为：
  - `SW = 0x0001`
  - `SW = 0x0002`
  - `SW = 0x0004`
- 每次变化都等待系统重新稳定，并检查：
  - `LED = SW | 0x10`
  - `dmem[0] = SW | 0x10`
  - `SEG = dmem[0] + dmem[1]`
  - `meip = 0`
  - `claimed = 0`

预期结果：

- 第 1 次中断后：`LED = 0x0011`，`SEG = 0x13`
- 第 2 次中断后：`LED = 0x0012`，`SEG = 0x14`
- 第 3 次中断后：`LED = 0x0014`，`SEG = 0x16`

判定标准：

- 三次中断都能独立完成一轮 `claim -> handler -> complete -> mret`
- 主程序每次都能恢复，并基于新的 `dmem[0]` 继续计算
- 不出现 `claimed` 残留或 `meip` 长期拉高

仿真结果：

```text
PASS: repeated switch interrupts passed led=0014 seg=16 mem0=00000014 mem2=00000016
```

测试结论：

- 同一开关外设已验证可连续 3 次触发中断
- 每次中断后 `claimed` 都能清零，`meip` 都能回落
- handler 对 `dmem[0]` 的修改会被主程序后续立即读到并继续计算
- 当前外部中断链路已具备基本重复响应能力

### 3.6 PS/2 键盘中断测试

测试目的：
验证 `io_ps2 -> PLIC 源 2 -> CPU trap -> PS2_RDATA 读清中断源 -> complete -> mret` 这条链路已经可用。

相关文件：

- 板级顶层：[`src/fpga_ps2_irq_top.v`](D:/project/RISC-V-cpu/26summer-CPU-design/src/fpga_ps2_irq_top.v)
- 程序映像：[`sim/ps2_irq_demo.mem`](D:/project/RISC-V-cpu/26summer-CPU-design/sim/ps2_irq_demo.mem)
- 仿真用例：[`sim/tb_ps2_irq_demo.v`](D:/project/RISC-V-cpu/26summer-CPU-design/sim/tb_ps2_irq_demo.v)

程序行为：

- 初始化阶段：
  - 配置 `PRIORITY[2] = 1`
  - 配置 `ENABLE = 0x0004`
  - 配置 `THRESHOLD = 0`
  - 写 `PS2_CTRL = 0x101`，打开 `enable` 与 `irq_en`
  - 打开 CPU 的 `mie.MEIE` 与 `mstatus.MIE`
- 主循环：
  - 从数据内存读取 `last_scan_code`
  - 从数据内存读取 `irq_count`
  - `SEG` 显示最近一次扫描码
  - `LED[15:8]` 显示中断次数，`LED[7:0]` 显示最近一次扫描码
- 中断处理程序：
  - 读 `CLAIM/COMPLETE`，记录 claim ID
  - 若 claim ID 不是 2，则把扫描码显示区写成 `EE`
  - 若 claim ID 是 2，则读 `PS2_RDATA` 清除 `valid`
  - 保存扫描码，`irq_count + 1`
  - 写 `CLAIM/COMPLETE = 2`
  - `mret` 返回主循环

仿真输入：

- 第一次发送扫描码 `0x1C`
- 第二次发送扫描码 `0x1B`

仿真结果：

```text
PASS: ps2 irq demo passed led=021b seg=1b mem_code=0000001b mem_count=00000002 claim=00000002
```

结果解释：

- `claim=2`：说明 PLIC 返回的中断源 ID 正确为 PS/2
- `mem_count=2`：说明 CPU 已成功处理两次 PS/2 中断
- `mem_code=0x1B`：说明最后一次读取到的扫描码正确写回软件可见状态
- `LED=0x021B`：高字节 `0x02` 表示已响应 2 次中断，低字节 `0x1B` 表示最近一次扫描码
- `SEG=0x1B`：七段数码管同步显示最近一次扫描码

板级验证方法：

- 使用 `fpga_ps2_irq_top` 作为综合顶层
- 将 `ps2_irq_demo.mem` 加入 Vivado 工程的 Memory Initialization Files
- 烧录后保持开发板复位释放、PS/2 键盘接入
- 上电空闲时预期：
  - `LED = 0000`
  - `SEG = 00`
- 依次按下键盘按键：
  - 按 `A`（make code `0x1C`）后，预期 `LED = 011C`，`SEG = 1C`
  - 按 `S`（make code `0x1B`）后，预期 `LED = 021B`，`SEG = 1B`
  - 按 `D`（make code `0x23`）后，预期 `LED = 0323`，`SEG = 23`

注意事项：

- 这里只统计按下事件。`F0` 释放前缀会被 `io_ps2` 硬件吞掉，所以松键不会单独显示
- `E0` 扩展前缀也会被硬件吞掉，因此更适合用字母键做基础验证
- 若数码管出现 `EE`，说明 claim 回来的中断源 ID 不是 2，应优先检查 PLIC 使能位和 PS/2 中断接线

## 4. 当前测试覆盖范围总结

本轮已经实际覆盖：

- 取指与主循环执行
- 数据存储器读写
- MMIO 地址译码
- LED / 数码管 / 开关访问

- CPU 可以执行程序并访问存储与外设

- CPU 可以进入 trap、执行 handler、再返回主程序
- 中断服务程序能够修改存储状态，并影响后续主程序结果

### 6.3 通用 SoC 板上验证现象

上电或复位释放后：

- `LED = 0000`
- 数码管显示 `03`

含义：

- 主程序正在执行 `dmem[0] + dmem[1]`
- 初始值为 `1 + 2 = 3`
- 说明 CPU 取指、普通 dmem 读写、cache 访问和 SEG MMIO 写入已经跑通

拨动 `SW[0] = 1` 后：

- `LED = 0011`
- 数码管显示 `13`

含义：

- `io_switches` 产生外部中断
- `interrupt_controller` 将中断送到 CPU 的 `meip`
- 当前 SoC 中 `sw_irq` 接在 `irq_sources[10]`，因此板测程序使能的是 PLIC 中断源 10
- CPU 跳转到 `mtvec = 0x100`
- handler 读取 `SW = 0x01`
- handler 写 `LED = 0x01 | 0x10 = 0x11`
- handler 写 `dmem[0] = 0x11`
- `mret` 返回后主程序重新计算 `0x11 + 0x02 = 0x13`

继续改变开关：

- `SW = 0002` 后，预期 `LED = 0012`，数码管显示 `14`
- `SW = 0004` 后，预期 `LED = 0014`，数码管显示 `16`
- `SW = 0008` 后，预期 `LED = 0018`，数码管显示 `1A`

判定标准：

- 每次拨动开关后 LED 都能更新为 `SW | 0x10`
- 数码管显示值等于 `(SW | 0x10) + 2`
- 不需要反复按 reset 才能刷新
- 多次触发后系统不死机

### 6.5 失败现象定位表

| 现象 | 优先怀疑位置 | 处理方式 |
| --- | --- | --- |
| 上电数码管不是 `03` | `board_test.mem` 未加载或 CPU 未运行 | 检查 Memory Initialization Files，重新综合实现 |
| 数码管是 `03`，拨动 SW 后 LED 不变 | SW 中断源或 PLIC 配置异常 | 检查 `io_switches`、`irq_sources`、`interrupt_controller` |
| LED 变为 `0011`，数码管不变 | handler 写内存后主循环未继续读到新值 | 回看 dmem/cache 访存路径 |
| LED 跟随变化，但数码管始终 0 | SEG MMIO 地址或 `io_seg7` 写入异常 | 检查 `0x80000010` 地址译码和 `seg_cs` |
| 必须 reset 后才更新 | 流水线停顿/flush 或取指响应异常 | 先跑 `tb_pipeline_mmio_mirror` 和 `tb_pipeline_soc_integration` |
| PS/2 中断次数增加但扫描码不变 | PS/2 RDATA 读取或 valid 清除异常 | 检查 `io_ps2` 和 handler 的读寄存器逻辑 |
| PS/2 单次按键无响应，连续按键才响应 | USB Host/键盘链路状态问题可能性较高 | 记录板载 USB Host 灯状态，换键盘或复测 |
