# 完整计算机系统验收演示原理说明

本文解释验收程序为什么能够证明系统功能，并给出对应汇编代码位置。现场操作和预期结果见 [项目验证手册](系统验收演示说明.md)。

## 1. 演示覆盖范围

同一份 331 条指令的 RV32I 程序覆盖：

1. 五级流水线 CPU 的算术、逻辑、移位、分支、跳转和访存。
2. 指令存储器、数据存储器和直接映射数据 Cache。
3. 开关、按钮、LED、八位数码管和 PS/2 的 MMIO。
4. PLIC 的 priority、enable、threshold、claim 和 complete。
5. CPU 机器外部中断入口、现场保存、外设清源和 `mret`。
6. cycle、instret、stall、flush、CPI 和吞吐量分析。

相关文件：

| 内容 | 文件 |
|---|---|
| 汇编源码 | `sim/system_acceptance_demo.asm` |
| 指令映像 | `sim/system_acceptance_demo.hex` |
| 自动验收 testbench | `sim/tb_system_acceptance_demo.v` |
| FPGA 验收顶层 | `src/fpga_acceptance_top.v` |
| Vivado 构建脚本 | `scripts/vivado_build_acceptance.tcl` |

## 2. 软件总体结构

### 2.1 初始化

汇编第 15–23 行清零开关、PS/2、按钮中断状态和 benchmark 标记；第 25–28 行只在中断启用前读取一次初始开关值。

第 30–59 行完成：

- `mtvec=0x800`；
- PLIC 源 2（PS/2）优先级为 2；
- 源 8（按钮）、源 10（开关）优先级为 1；
- 使能掩码为 `0x504`；
- threshold 为 0；
- 打开 `mie.MEIE` 和 `mstatus.MIE`。

### 2.2 页面分派不是开关轮询

主循环第 61–83 行读取 DMEM `0x200` 中的开关快照，然后根据高 4 位分派页面。它没有反复读取开关 MMIO。

开关 MMIO `0x80000008` 只在 ISR 第 375–382 行读取。ISR 将新值保存到 `0x200`，主循环随后处理该状态。因此程序采用“中断更新快照、主循环处理事件”的结构，而不是轮询硬件开关。

### 2.3 统一输出

`write_outputs` 位于第 310–338 行：

1. 把 `x11` 写入 LED MMIO `0x8000000C`。
2. 将 `x10` 从低到高每次右移 4 位。
3. 分别写入八个数码管 DIGIT 寄存器。
4. 通过 `jalr` 返回调用页。

按钮使用 `0x80000034`，因为 `0x80000030` 已由 DIGIT7 使用。

## 3. 各验证页面原理

### 页面 0：CPU 自检

对应汇编第 85–123 行。

第 87–99 行执行加、减、异或和左移，并用条件分支逐项比较结果；相邻指令连续消费结果，同时验证 EX/MEM、MEM/WB 前递。第 101–108 行执行 DMEM 写后读和有符号分支。

任意比较失败都显示 `DEADC0DE`；全部通过才由第 110–116 行构造 `600DC0DE` 并点亮 LED。因此开关为 `0000` 时显示的不是开关值，而是自检成功签名。

### 页面 1：MMIO 镜像

对应汇编第 125–131 行。

ISR 保存的完整 16 位开关快照与 `0x10000000` 合成数码管内容，同时原值直接送往 LED。开关值改变时显示同步改变，证明 CPU 经地址译码器实际读写外设寄存器，而不是固定常数输出。

### 页面 2：ALU 与流水线前递

对应汇编第 133–146 行。

程序连续执行：

```asm
andi → addi → xori → slli → andi
```

“按位”表示 `xori` 对各二进制位分别异或、`andi` 对各位分别屏蔽。“前递”表示下一条指令不等待上一条写回寄存器堆，而直接取得 EX/MEM 或 MEM/WB 中的新结果。例如第 136 行产生 `x+3`，第 137 行立即使用；前递错误会直接造成 `YY` 不匹配。

### 页面 3：DMEM 与 Cache

对应汇编第 148–185 行。

第 150–157 行把 `p..p+3` 写入同一 Cache 行；第 158–164 行连续读回并求和。Cache 为 16 行、每行 4 字、直接映射、写直达、写不分配。

一行共 16 B，整个 Cache 的索引跨度为：

```text
16 行 × 4 字 × 4 B = 256 B = 0x100
```

所以 `0x000` 与 `0x100` index 相同、tag 不同。第 167–172 行在两者间切换，强制发生冲突替换和重填。任何数据错误进入第 181–185 行的失败路径。

Cache 位于 CPU 数据总线出口，只缓存普通内存地址；MMIO 旁路 Cache，避免外设访问产生错误命中。

### 页面 4：Fibonacci 小程序

对应汇编第 187–206 行。

`x12` 保存输入 `n`，`x13/x14` 保存相邻 Fibonacci 项，`x15` 保存循环次数。每轮使用 `beq` 判断结束、更新迭代状态，再用 `jal x0` 返回循环头。

该页面不是单条指令测试，而是完整验证循环、taken/not-taken 分支、状态迭代和连续数据相关。

### 页面 5：流水线性能压力

对应汇编第 208–260 行。

四段 workload 分别是：

1. 第 215–220 行：连续 ALU 相关，观察前递。
2. 第 226–231 行：`lw` 后立即消费，制造 load-use hazard。
3. 第 236–239 行：连续 taken branch，制造流水线 flush。
4. 第 245–251 行：交替读取 `0x000/0x100`，制造 Cache 冲突。

第 210–254 行在各段边界向 DMEM `0x380` 写 `0x51–0x55`。testbench 在写握手时采样 cycle、instret、stall、flush，并用相邻采样值相减，得到每段独立指标。

### 页面 6：开关中断

页面显示代码位于第 262–272 行，开关 ISR 位于第 375–382 行。

硬件和软件链路：

```text
SW 管脚
→ 两级同步器
→ sw_sync != sw_prev
→ irq_pending
→ PLIC 源 10
→ meip
→ CPU trap
→ mtvec=0x800
→ claim=10
→ 读取开关 MMIO 清源
→ complete
→ mret
```

读取开关后，ISR 将值写入 `0x200`，并把 `0x204` 的中断次数加一。页面显示最后 claim ID 和累计次数，从板上现象可以确认整个中断闭环。

### 页面 7：PS/2 中断

页面代码位于第 274–284 行，ISR 位于第 384–392 行。

PLIC claim ID 2 后，ISR 读取 PS/2 DATA，读取动作清除外设 pending；随后保存低 8 位扫描码到 `0x20C`，并增加 `0x210` 的计数。页面读取这两个软件状态生成显示。

### 页面 8：按钮中断

页面代码位于第 286–298 行，ISR 位于第 394–402 行。

顶层把五个按钮映射为：

```text
btn[4:0] = {BTNC, BTNU, BTNR, BTND, BTNL}
```

`io_buttons` 使用两级同步器，只在 `btn_sync & ~btn_prev` 非零时置 pending，因此按下触发、松开不触发。PLIC 将按钮映射为源 8。

ISR 读取 `0x80000034`，读取动作清除 pending；随后将按钮位图保存到 `0x214`，将 `0x218` 计数加一。页面显示 `88BBCCCC`，LED5 是页面标志，LED[4:0] 是最近按钮位图。

## 4. 中断处理程序公共流程

ISR 从第 342 行开始：

1. 第 343–360 行保存主程序可能使用的寄存器。
2. 第 362–365 行读取 PLIC CLAIM 并记录 claim ID。
3. 第 367–373 行按 ID 分派：10 为开关、2 为 PS/2、8 为按钮。
4. 各外设处理分支读取设备寄存器以清除外设 pending。
5. 第 404–406 行把 claim ID 写回 COMPLETE。
6. 第 408–426 行恢复寄存器。
7. 第 427 行执行 `mret`，恢复中断前 PC 和中断使能状态。

CLAIM 解决“当前处理哪个源”，外设读取解决“设备为什么仍在请求”，COMPLETE 告诉 PLIC 本次服务结束，三者缺一都会导致重复进入或中断卡住。

## 5. 性能量化

### 5.1 仿真结果

| 区间 | cycle | instret | CPI | stall | flush |
|---|---:|---:|---:|---:|---:|
| ALU/前递 | 303 | 84 | 3.60 | 204 | 15 |
| load-use | 468 | 84 | 5.57 | 369 | 15 |
| taken branch | 204 | 51 | 4.00 | 138 | 15 |
| Cache 冲突 | 456 | 52 | 8.76 | 397 | 7 |

计算方法：

```text
CPI = Δcycle / Δinstret
吞吐量 = 时钟频率 / CPI
```

ALU 区间 CPI 最低；load-use 因等待访存数据而增加 stall；taken branch 引发错误路径 flush；直接映射 Cache 冲突同时包含重填和等待，因此 CPI 最高。

### 5.2 板载频率与静态时序

验收顶层使用 `CLK_DIV_BITS=18`：

```text
Fdemo = 100 MHz / 2^18 ≈ 381.47 Hz
```

该频率是为了便于肉眼观察，不代表 CPU 极限频率。

Vivado 2018.3 对 CPU 域施加 10.000 ns 评估约束后得到：

```text
WNS = -0.090 ns
TNS = -0.970 ns，共 16 个端点
未约束最大延迟端点 = 0
```

因此不能宣称 100 MHz 已通过。按关键路径约 10.090 ns 估算临界 Fmax 为 99.1 MHz；板载 381 Hz 远低于该值。最差路径从 EX/MEM 地址结果到 Cache 重填缓冲，约 72% 是布线延迟。

按 99.1 MHz 临界估计值计算：

| 区间 | CPI | 估算吞吐量 |
|---|---:|---:|
| ALU/前递 | 3.60 | 27.53 MIPS |
| load-use | 5.57 | 17.79 MIPS |
| taken branch | 4.00 | 24.78 MIPS |
| Cache 冲突 | 8.76 | 11.31 MIPS |

实现资源为 3423 LUT（5.40%）、5134 个寄存器（4.05%）、8 个 Block RAM Tile（5.93%）、0 个 DSP。

## 6. 性能瓶颈与优化方向

| 现象 | 原因 | 优化方向 |
|---|---|---|
| load-use stall 较多 | load 数据到达时机晚 | load 数据提前旁路、降低存储延迟 |
| taken branch 产生 flush | 分支在后级确定 | 提前分支判定、增加分支预测 |
| Cache 冲突 CPI 最高 | 直接映射同 index 反复替换 | 增加相联度、容量或预取 |
| 100 MHz 差 0.090 ns | Cache 重填路径布线较长 | 重填路径流水化、物理综合和布局约束 |
| 板上观察速度慢 | 主动使用 18 位分频 | 功能展示完成后切换更高 CPU 时钟 |

仓库没有可用于同源对比的基础单周期 CPU，因此 `CPI≈1` 只能作为理论参照，不能虚构单周期 Fmax 数据。

## 7. 验收总结

该演示不是把开关直接连接到 LED。CPU 实际运行 331 条指令组成的程序；开关和按钮分别通过 PLIC 源 10 和源 8 更新软件状态；主程序执行 ALU、DMEM/Cache、Fibonacci 和性能压力任务；结果再经 MMIO 输出。自动仿真同时量化 cycle、instret、stall、flush 和 CPI，从功能、系统集成与性能三个层面覆盖进阶层次验收要求。
