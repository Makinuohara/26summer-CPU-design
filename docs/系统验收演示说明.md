# 进阶层次完整计算机系统验收演示说明

## 1. 演示目标

本演示用同一套 RV32I 汇编程序依次证明以下功能已经形成完整、可运行的计算机系统：

1. 五级流水线 CPU 能正确执行算术、逻辑、移位、分支、跳转和访存指令。
2. 指令存储器能加载并连续运行一个超过简单 smoke test 的程序。
3. 数据存储器与直接映射 Cache 能完成写入、命中读取和冲突重填。
4. CPU 能通过 MMIO 访问拨码开关、五个按钮、LED、八位数码管和 PS/2 控制器。
5. PLIC 能完成外设中断仲裁，CPU 能执行 `claim → ISR → complete → mret`。
6. 性能计数器能量化 cycle、instret、stall、flush、CPI 和吞吐量。

板上主流程使用拨码开关选页，以 LED 和数码管显示结果，并用五个实体按钮验证第二类边沿中断。PS/2 键盘是加分项，键盘不可用时不影响 CPU、存储、Cache、按钮和 PLIC 的主验收结论。

## 3. 操作规则

- `SW[15:12]`：功能页编号，范围 0–8；9–F 显示 `F000000x`，用于提示页号无效。
- `SW[11:0]`：该页面的输入数据。
- 数码管最左一位显示页编号，后续位显示输入或计算结果。
- 每次拨动开关都会触发 PLIC 源 10 中断。ISR 读取开关快照并返回，主程序再根据快照执行对应页面。
- 五个按钮依次对应 `BTNL/BTND/BTNR/BTNU/BTNC = bit[0..4]`，按下触发 PLIC 源 8，松开不触发。
- 页面切换后等待数码管稳定再讲解。CPU 为方便观察被分频到约 381 Hz，复杂页面可能需要短暂等待。

建议验收前把所有开关拨到 0，按下低有效复位键后释放。

## 4. 逐步验收流程

### 步骤 1：上电自检——CPU、流水线和存储基本通路

操作：

1. 将 `SW=0x0000`。
2. 复位并释放。
3. 等待数码管稳定。

预期：

```text
SEG = 600DC0DE
LED = FFFF
```

运行逻辑：复位后先清零中断计数和测试标记（汇编第 15–23 行），设置 `mtvec`、PLIC 优先级/使能和机器外部中断（第 30–59 行），主循环再从开关快照取出高 4 位分派页面（第 61–83 行）。页面 0 依次执行加、减、异或、移位、比较、条件分支以及 DMEM 写后读（第 85–108 行）。任何比较失败都跳到失败签名；全部通过才构造 `600DC0DE` 并点亮全部 LED（第 110–123 行）。因此 `SW=0000` 显示该值并非开关内容，而是 CPU 自检成功签名。

### 步骤 2：MMIO 镜像——输入、地址译码和输出

操作：依次拨成 `SW=0x1123`、`0x1A5A`。

预期：

| SW | SEG | LED |
|---|---|---|
| `1123` | `10001123` | `1123` |
| `1A5A` | `10001A5A` | `1A5A` |

运行逻辑：开关变化首先由 ISR 的第 375–382 行读取 `0x80000008`，保存到 DMEM `0x200` 并累加中断计数；主循环第 61–68 行分派到页面 1。页面代码第 125–131 行把快照与页头 `0x10000000` 合成显示值，同时把完整 16 位开关值传给 LED。统一输出函数第 310–338 行写 LED，再从最低半字节开始把 32 位结果拆成八个十六进制数字写入数码管 MMIO。输入和输出一一对应，证明不是硬编码显示。

### 步骤 3：ALU 数据通路——可复核的实时计算

本页取低 8 位 `x`，计算：

```text
y = (((x + 3) XOR 0x5A) << 1) AND 0xFF
```

数码管格式为 `2000XXYY`，LED 低 8 位显示 `YY`。

| SW | x | 预期 y | SEG | LED |
|---|---:|---:|---|---|
| `2012` | `12` | `9E` | `2000129E` | `009E` |
| `2020` | `20` | `F2` | `200020F2` | `00F2` |

运行逻辑（汇编第 133–146 行）：`andi` 取出 `x`，随后 `addi → xori → slli → andi` 连续产生并消费中间结果，最后拼装显示字并输出。这里的“按位”指 `xori` 对 32 个二进制位分别异或、`andi` 对各位分别屏蔽；“前递”指流水线不等结果写回寄存器堆，而把 EX/MEM 或 MEM/WB 级刚产生的结果直接送到下一条指令的 EX 输入。例如第 136 行刚得到 `x+3`，第 137 行的 `xori` 紧接着就使用它；若前递错误，显示的 `YY` 会立即不匹配。

### 步骤 4：DMEM 与 Cache——写入、命中和冲突重填

操作：设置 `SW=0x3005`。

程序行为（汇编第 148–185 行）：

1. 向地址 `0x00/0x04/0x08/0x0C` 写入 `5/6/7/8`。
2. 连续读回同一 Cache 行并求和，得到 `0x1A`。
3. 访问同 Cache 索引、不同 tag 的 `0x100`，制造直接映射冲突。
4. 再次读取地址 0，验证冲突重填后的数据仍正确。

预期：

```text
SEG = 3000001A
LED = FFFF
```

运行逻辑：第 150–157 行把 `p..p+3` 写到连续四个字；写策略是写直达、写不分配，因此后端 DMEM 始终得到新值。第 158–164 行连续读取并求和，第一次读可能重填，后续同一行读取命中。16 行 × 4 字 × 4 B = 256 B，所以地址 `0x000` 和 `0x100` 的 index 相同而 tag 不同；第 167–172 行访问 `0x100` 后再访问 `0x000`，强制发生替换和再次重填。最后比较原值、冲突地址值与 checksum，任意错误显示 `3BAD0000`，全部正确才显示 `3000001A`（第 173–185 行）。Cache 位于 CPU 的数据总线出口，只缓存普通内存；MMIO 直接旁路 Cache，避免外设读写被缓存。

### 步骤 5：小型程序——Fibonacci 完整运行

操作：设置 `SW=0x400A`，即计算 `fib(10)`。

预期：

```text
SEG = 400A0037
LED = 0037
```

其中十六进制 `0x37` 等于十进制 55。

运行逻辑（汇编第 187–206 行）：`x12=n`，`x13/x14` 保存相邻两项，`x15` 是循环计数；每轮先用 `beq` 判断结束，再计算下一项并更新状态，通过 `jal x0` 回到循环头。结束后把页号、输入 `n` 和 `fib(n)` 拼成八位显示。该页同时覆盖循环控制、taken/not-taken 分支、迭代状态和连续数据相关，证明系统能运行有实际控制流的小型程序。

### 步骤 6：流水线压力程序——性能瓶颈

操作：设置 `SW=0x5000`。

程序在汇编第 208–260 行依次运行：

1. ALU/前递循环。
2. `lw` 后立即使用的 load-use 循环。
3. 连续 taken branch 循环。
4. 地址 0 与 `0x100` 交替读取的 Cache 冲突循环。

每段开始前分别向 DMEM `0x380` 写标记 `0x51–0x55`（第 210–212、222–223、233–234、241–242、253–254 行）。testbench 在标记写握手时同时采样 cycle、instret、stall、flush，再作相邻差值；因此区间边界由真实总线事务确定，不依赖人工数周期。LED 低五位全亮表示各阶段完成，数码管显示 `5` 开头的 checksum。

当前自动仿真的一组结果：

| 区间 | cycle | instret | CPI | stall | flush |
|---|---:|---:|---:|---:|---:|
| ALU/前递 | 303 | 84 | 3.60 | 204 | 15 |
| load-use | 468 | 84 | 5.57 | 369 | 15 |
| taken branch | 204 | 51 | 4.00 | 138 | 15 |
| Cache 冲突 | 456 | 52 | 8.76 | 397 | 7 |

不同 RTL 版本修改后应重新运行仿真，以新输出为准。

### 步骤 7：PLIC 和外部中断

操作：设置 `SW=0x6000`，随后轻拨任意低位开关再拨回。

数码管格式：

```text
6 00A CCCC
```

- `A` 表示最近一次 claim ID 为十进制 10，即十六进制 `A`。
- `CCCC` 是累计开关中断次数，会随每次拨动增加。
- LED0 始终亮，表示 ISR 返回后主循环仍在运行。

运行逻辑：初始化第 35–49 行把源 10 设为优先级 1，并以掩码 `0x504` 同时使能源 2、8、10；第 55–59 行打开 `mie.MEIE` 和 `mstatus.MIE`。开关同步器检测变化 → 源 10 pending → PLIC 按 priority/enable/threshold 拉高 `meip` → CPU 记录 `mepc/mcause`、清空错误路径并跳到 `mtvec=0x800`。ISR 第 342–373 行保存现场并 claim，第 375–382 行读取开关以清源、保存快照并计数，第 404–406 行 complete，最后第 408–427 行恢复现场并 `mret`。页面 6 的显示代码在第 262–272 行读取计数与最后 claim ID，所以能直观看到整个链路确实走完。

### 步骤 8：PS/2 键盘中断（可选加分）

操作：

1. 接好 PS/2 键盘后设置 `SW=0x7000`。
2. 依次按 `A` 和 `S`。

数码管格式为 `7-中断次数-扫描码`。典型扫描码：

| 按键 | 扫描码 |
|---|---|
| A | `1C` |
| S | `1B` |

程序通过 PLIC 源 2 接收键盘中断。ISR 第 384–392 行读取 PS/2 DATA 清 pending、保存扫描码并累加计数；页面 7 第 274–284 行组合显示。若现场键盘桥接链路首次输入不稳定，先连续按两次再观察；仍不可用时立即回到页面 6，以已通过的自动 PS/2 仿真作为证据，不让可选外设影响主验收。

### 步骤 9：五个实体按钮与第二类中断源

操作：设置 `SW=0x8000`，依次按下并松开 `BTNL、BTND、BTNR、BTNU、BTNC`，最后可同时按多个按钮。

数码管格式为：

```text
88BBCCCC
```

- `BB` 是最近一次按下时读取的 5 位按钮位图：左/下/右/上/中分别为 `01/02/04/08/10`。
- `CCCC` 是按钮按下中断累计次数；每按下一次增加 1，单纯松开不会增加。
- LED5 恒亮作为页面标志，LED[4:0] 同步显示最近按钮位图。例如按中键显示 LED=`0030`，组合 `BTNC+BTNR+BTNL` 显示 `0035`。

运行逻辑：实体管脚先在 `fpga_acceptance_top` 中拼成 `{BTNC,BTNU,BTNR,BTND,BTNL}`。`io_buttons` 用两级触发器同步输入，只对 `btn_sync & ~btn_prev` 的上升沿置 pending；PLIC 把它作为源 8。ISR claim 到 8 后执行汇编第 394–402 行，从 `0x80000034` 读取按钮值（该读操作同时清 pending），保存到 DMEM `0x214` 并累加 `0x218`；随后共用 complete/恢复路径返回。页面 8 第 286–298 行读取这两个状态并显示。因此现场可分别验证实体管脚、同步器、MMIO、PLIC 仲裁、ISR 和输出闭环。

> 地址说明：八位数码管的 DIGIT7 已占用 `0x80000030`，所以按钮寄存器使用下一个字地址 `0x80000034`；这避免按钮与最左数码管互相覆盖。

## 5. 性能量化方法

### 5.1 板载演示频率

`clk_div` 输出计数器最高位，当前 `CLK_DIV_BITS=18`：

```text
Fdemo = 100 MHz / 2^18 ≈ 381.47 Hz
```

这是为了让人眼能稳定观察程序状态而主动选择的演示频率，不是流水线 CPU 的极限频率。

### 5.2 CPI 与吞吐量

对每个 benchmark 区间取性能计数器差值：

```text
CPI = Δcycle / Δinstret
实现极限吞吐量 = Fmax / CPI
板载演示吞吐量 = 381.47 / CPI  指令/秒
```

例如当前 Cache 冲突区间 `CPI=8.76`，板载分频下约为：

```text
381.47 / 8.76 ≈ 43.5 指令/秒
```

验收构建额外使用 `constraints/acceptance_cpu_timing.xdc`，在不改变板上 18 位分频硬件的前提下，以 100 MHz 目标约束 CPU 时钟域，避免 CPU 路径被静态时序分析遗漏。本次 Vivado 2018.3 实现后结果为：

```text
CPU 目标周期 = 10.000 ns
CPU WNS      = -0.090 ns
TNS          = -0.970 ns（16 个端点）
未约束最大延迟端点 = 0
```

因此不能宣称“100 MHz 已通过”，但只差 0.090 ns，按关键路径约 `10.090 ns` 估算临界 Fmax 为 `99.1 MHz`。失败路径从 CPU 的 EX/MEM 地址结果到 Cache 重填缓冲，其中布线延迟约占 72%，后续可通过 Cache 重填路径加寄存器、物理综合或布局约束优化。板载验收使用约 381 Hz 的分频时钟，远低于该临界值，不影响当前 bitstream 的现场功能验证。

按时序临界估计值 99.1 MHz 计算理论吞吐量（这是估算值，不是已验证工作频率）：

| 区间 | CPI | 99.1 MHz 估算吞吐量 |
|---|---:|---:|
| ALU/前递 | 3.60 | 27.53 MIPS |
| load-use | 5.57 | 17.79 MIPS |
| taken branch | 4.00 | 24.78 MIPS |
| Cache 冲突 | 8.76 | 11.31 MIPS |

实现后资源占用为 3423 个 LUT（5.40%）、5134 个寄存器（4.05%）、8 个 Block RAM Tile（5.93%）、0 个 DSP。现场应先报告确定的 381 Hz 演示频率和 CPI，再把 99.1 MHz 明确称为静态时序估计值。

### 5.3 瓶颈和优化建议

| 现象 | 原因 | 优化方向 |
|---|---|---|
| load-use CPI 上升 | load 数据到达前相关指令必须暂停 | 更低延迟存储、load 数据提前旁路 |
| 分支产生 flush | 分支在后级确定，错误路径指令被清空 | 提前分支判定、静态或动态分支预测 |
| Cache 冲突 CPI 最高 | 直接映射同索引地址反复替换 | 增加容量/相联度、预取、增加 hit/miss 计数器 |
| 板上运行较慢 | 为可观察性使用 18 位分频 | 完成时序收敛后减小分频或直接使用更高 CPU 时钟 |

仓库当前没有可用的基础单周期 CPU RTL，因此不能给出同源、同工具、同程序的单周期 Fmax 对比。单周期 `CPI≈1` 只能作为理论参照，验收中应明确这一限制，不使用虚构数据。

## 6. 现场故障预案

| 现象 | 检查和恢复 |
|---|---|
| 数码管无显示、LED 全灭 | 确认烧录的是 `system_acceptance_demo.bit`；将 SW 清零并重新复位 |
| 显示不是 `600DC0DE` | 确认 hex 被加入 Memory Initialization Files，检查 Vivado 日志中的 `$readmemh` 路径 |
| 页面切换较慢 | 等待当前 ISR 和八次数码管 MMIO 写入完成，不要连续快速拨动 |
| 页面 6 计数不是固定值 | 正常；之前每次切页本身都会产生一次开关中断，只检查计数递增和 claim ID 为 A |
| 页面 8 按键无计数 | 确认使用验收顶层且五个 BTN 管脚已约束；按下后应看到 claim ID 8，读取地址必须是 `0x80000034` |
| 按住按钮只计数一次 | 正常；按钮采用上升沿触发，必须松开后再次按下才产生下一次中断 |
| PS/2 无响应 | 连续输入两次；仍失败则使用页面 6 和自动仿真完成主验收 |
| 老师询问 Cache 是否真的命中 | 展示 testbench 的四个性能区间，并指出冲突访问 CPI 明显高于普通 ALU 和 load-use 区间 |

## 7. 一分钟总结话术

“这不是把 LED 直接接到开关，而是 CPU 运行 331 条指令组成的汇编映像。开关和五个按钮分别作为 PLIC 源 10 和源 8 进入 CPU，ISR 保存现场、claim、读取 MMIO 清源、complete 后再 mret；主程序还能执行 ALU、存储、Cache 冲突和 Fibonacci 等任务，最后通过 MMIO 驱动 LED 和八位数码管。仿真同时读取流水线内部 cycle、instret、stall、flush 计数，计算不同 workload 的 CPI。结果表明 load-use、分支清空和直接映射 Cache 冲突是主要瓶颈，对应优化方向分别是提前旁路、分支预测和提高 Cache 相联度。”
