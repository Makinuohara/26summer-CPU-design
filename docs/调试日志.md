# 调试日志

## Nexys 4 DDR 无法被 Vivado 识别问题

### 问题现象

在 Vivado 2018.3 的 Hardware Manager 中连接 Nexys 4 DDR 时，最开始只能看到 `localhost` 硬件服务器，无法打开具体 FPGA 设备。界面中出现过以下现象：

```text
No hardware target is open
localhost (0)
No hardware targets exist on the server [localhost:3121]
```

在安装 Digilent 驱动并调整跳线后，Vivado 能看到 Digilent 下载目标，但仍然提示 JTAG 链上没有 FPGA 设备：

```text
localhost (1)
xilinx_tcf/Digilent/...
No devices detected on target
```

这说明问题分成两个阶段：

```text
第一阶段：电脑没有正确识别 Digilent USB-JTAG 下载器
第二阶段：电脑识别到了下载器，但 JTAG 链没有扫到 FPGA 芯片
```

### 排查过程

1. 检查 Vivado Hardware Manager

   先确认 `hw_server` 是否启动，以及 `localhost` 是否连接。发现 `localhost` 可以连接，但最初目标数为 0，说明 Vivado 本身启动正常，问题更可能在 USB 设备、驱动或板卡连接链路。
2. 检查 Nexys 4 DDR 跳线

   根据 Digilent 入门文档和老师提示，重点检查 JP1 和 JP2：

   ```text
   JP1 / MODE：需要设置到 JTAG，才能通过 Vivado Hardware Manager 下载 bitstream
   JP2：与 SD/USB 启动相关，本次 JTAG 下载不是主要矛盾
   ```

   一开始 JP1 处于 QSPI 模式，此时开发板会运行 Flash 中的出厂程序，七段数码管出现类似“贪吃蛇”的移动光效。切换到 JTAG 后，出厂程序不再自动运行，只剩电源指示灯亮，这是正常现象，并不代表板子损坏。
3. 安装/重装 Digilent 相关驱动

   根据 Vivado 和 Digilent 的说明，安装 Digilent Adept Runtime，并检查 Windows 设备管理器中是否存在异常 USB 设备。过程中曾出现：

   ```text
   未知 USB 设备（设备描述符请求失败）
   ```

   这类提示说明 Windows 已经检测到有 USB 设备插入，但无法正确读取设备描述符，问题可能来自驱动、USB 口、USB 线或板卡 USB 接口。
4. 对比不同 USB 口和连接方式

   更换过电脑 USB 口，并反复执行 Vivado 中的 `Open Target`、`Auto Connect`、`Refresh Server` 等操作。虽然连接状态有变化，但仍然不能稳定识别 FPGA 设备，因此排除了单纯 Vivado 操作步骤错误的可能。
5. 对比其他小组的成功现象

   其他小组能够正常下载流水灯 bitstream，说明同一型号开发板和 Vivado 流程本身是可行的。于是把重点转向本地连接链路，包括 USB 线材、供电和数据传输能力。

### 根本原因

最终发现问题出在 Nexys 开发板使用的原装 Micro-USB 线。该线在当前环境下可以给开发板供电，但无法稳定进行 USB 数据传输，导致 Vivado 无法通过 Digilent USB-JTAG 链路识别 FPGA。

换用一根确认支持数据传输的 Micro-USB 线后，Vivado 能够正常识别开发板，后续可以进行 bitstream 下载验证。

### 解决方法

```text
1. 使用支持数据传输的 Micro-USB 线连接 Nexys 4 DDR 的 PROG/UART 接口
2. 确认电源开关打开
3. 确认 JP1 / MODE 设置为 JTAG
4. 打开 Vivado Hardware Manager
5. 选择 Open Target -> Auto Connect
6. 识别到 FPGA 后，选择 Program Device 下载 bitstream
```

### 验证结果

更换 USB 线后，可以继续进行板卡验证。先使用独立的流水灯 bitstream 验证下载链路：

```text
build/bitstreams/led_chaser_top.bit
```

流水灯能够运行，说明：

```text
USB 数据线正常
Digilent 驱动正常
Vivado Hardware Manager 能够识别并配置 FPGA
JTAG 下载链路正常
开发板基本硬件状态正常
```

随后可下载第一层 CPU bitstream：

```text
build/bitstreams/fpga_top.bit
```

通过拨码开关观察 CPU 测试程序结果：

```text
SW[1:0] = 01：观察 x5，期望为 0001
SW[1:0] = 10：观察 data_mem[0]，期望为 000c
```

### 经验总结

本次问题的关键不是 Verilog 代码或 bitstream 生成错误，而是硬件下载链路中的 USB 数据传输失败。Micro-USB 线即使能供电，也不一定支持或稳定支持数据通信。遇到 Vivado 无法识别 FPGA 时，应按以下顺序排查：

```text
1. 板卡供电是否正常
2. JP1 是否处于 JTAG 模式
3. Windows 设备管理器是否识别到 Digilent/USB Serial 设备
4. Vivado Hardware Manager 是否能看到 hw_server 和 target
5. USB 线是否支持数据传输
6. Digilent/Vivado cable driver 是否正确安装
```

其中 USB 线材问题最容易被忽略，因为“能供电”会让人误以为线一定没有问题。

## CPU + MMIO 联调显示异常问题

### 问题现象

在完成 `CPU + imem + dmem_bus_decoder + LED/SW/SEG` 联调并烧录上板后，最开始出现过以下现象：

```text
1. LED 和数码管最初都不响应 SW
2. 后续 LED 可以变化，但数码管始终显示 0
3. 再继续修改后，板上又出现 LED 熄灭、数码管全 0 的情况
```

表面上看像是 I/O 模块或顶层连接问题，但由于 LED、SEG、SW 三个外设并不是同时失效，所以更怀疑是 CPU 的执行路径、取指握手或测试程序本身有问题。

### 排查过程

1. 先检查顶层 MMIO 通路

   确认 `fpga_top` 中 `dmem_bus_decoder`、`io_switches`、`io_leds`、`io_seg7` 的片选和连接关系正确，排除明显接线错误。
2. 从上板观察切换到仿真定位

   仅靠 LED 和数码管现象很难判断到底是哪一级出错，因此转到 Vivado 仿真。这里我借助 AI 工具一起梳理了需要重点观察的信号，包括：

   ```text
   pc
   if_id_valid / id_ex_valid / mem_wb_valid
   dmem_addr / dmem_wdata
   x2 寄存器值
   ```

   同时补了 `tb_pipeline_mmio_mirror.v`，专门验证 `lw SW -> sw LED -> sw SEG` 这条最小 MMIO 路径。
3. 发现取指等待不应冻结整个后端流水线

   仿真中发现，当前 CPU 在 `imem_wait` 时把后端流水线也一起停住，导致已经进入 EX/MEM/WB 的老指令推进异常。于是把停顿条件拆开：

   ```text
   pipeline_stall = dmem_wait || load_use_stall
   perf_stall     = imem_wait || pipeline_stall
   ```

   这样取指等待只影响前端，不再错误阻塞后端提交。
4. 发现跳转后旧取指响应被错误写回 IF/ID

   仿真继续显示 `jal` 回跳后 CPU 仍可能取到旧 PC 对应的返回指令。为此在 CPU 中增加了 `discard_imem_resp`，在控制流重定向时丢弃飞行中的旧取指响应，避免错误路径指令重新进入流水线。
5. 发现 demo 程序本身也有问题

   为了测试 load 写回，我之前在 demo ROM 中插入了两个 `nop`，但后面的 `jal` 偏移没有同步修改，导致程序回跳位置错误，没有真正重新执行 `lw SW`。这一点也是在 AI 辅助下，通过仿真时序一起定位出来的。
6. 修正板级 demo IMEM 时序

   `fpga_demo_imem.v` 原本把 `imem_ack` 和 `imem_data` 在同一拍寄存输出，和正式 `imem.v` 的响应行为不一致，容易放大取指时序问题。后续把它改成和正式 `imem` 一致的响应方式，再继续验证。

### 根本原因

这次问题不是单一模块故障，而是三个因素叠加：

```text
1. CPU 取指等待逻辑过于保守，错误冻结后端流水线
2. 控制流重定向时，没有丢弃旧的取指返回
3. 板级 demo 程序插入 nop 后，jal 回跳偏移没有同步修正
```

因此板上才会出现“有时 LED 可变、有时数码管全 0、行为前后不一致”的现象。

### 解决方法

```text
1. 修改 pipeline_cpu_top.v，拆分 imem_wait 与真正的 pipeline_stall
2. 在 CPU 中加入 discard_imem_resp，丢弃跳转后的旧取指响应
3. 修改 fpga_demo_imem.v 的返回时序，使其与正式 imem 行为一致
4. 修正 demo 程序中 jal 的回跳偏移
5. 通过 mmio 定向 testbench 反复验证后，再重新综合烧录
```

### 验证结果

仿真最终通过：

```text
SW = 3 -> LED = 0003, SEG = 03
SW = 2 -> LED = 0002, SEG = 02
PASS: MMIO mirror pipeline path passed
```

重新烧录后，板上 LED 和数码管也已经能够正常随开关变化，说明：

```text
CPU 的 lw/sw 基本执行路径正确
MMIO 地址译码正确
LED/SEG 外设写通路正确
当前板级 demo 可以作为后续中断联调的基础版本
```

### 经验总结

这次调试说明，SoC 联调里“显示异常”不一定是显示模块自己的问题，很多时候真正的问题在 CPU 流水线控制或测试程序本身。后续再遇到类似问题，优先按下面顺序排查：

```text
1. 先确认测试程序是否正确
2. 再确认 MMIO 地址译码和顶层连接
3. 最后用仿真看 CPU 内部推进、写回和跳转逻辑
```

另外，这次使用 AI 工具的主要帮助不是直接替我写代码，而是协助我把问题拆成“顶层连接、仿真信号、流水线停顿、控制流重定向、测试程序”几个层次逐步定位，明显提高了排错效率。

## PS/2 键盘中断输入异常问题

### 问题背景

这个问题最早不是在 CPU 联调时暴露出来的，而是王浩然同学在单独测试 I/O 模块时先遇到的。

当时的现象是：

```text
键盘连续多次输入后，偶尔能得到正确扫描码
但单次按键、少量按键时，经常没有反应
```

由于板上 USB Host 侧 `BUSY` 灯会闪烁，而且不同键盘、不同按键速度下表现不完全一致，最开始大家普遍认为这是：

```text
1. 键盘兼容性问题
2. USB/PS2 连接问题
3. 板上输入不稳定
```

因此在这个阶段，问题还没有被明确归因到 RTL 逻辑。

### 统一联调时再次复现

后来在王哲和黄严做 `CPU + memory + IO + interrupt_controller` 统一联调时，这个问题再次出现，而且比单测阶段更容易观察：

```text
1. 烧录后空闲时，LED = 0000，SEG = 00
2. 单次按键时，往往没有任何反应
3. 连续快速按同一个键时，后几次输入又能显示正确扫描码
4. 在更早的一版程序里，第一次按键还会显示 AA
```

这说明问题不再只是“键盘有时不工作”，而是存在一个更具体的行为规律：

```text
第一次输入和后续输入并不等价
空闲一段时间后的第一次输入最容易失败
连续输入时，后续输入反而更容易成功
```

### 板上现象进一步分析

在板级观察中，我们注意到一个非常关键的现象：

```text
1. 开始时 BUSY 灯处于闪烁状态
2. 第一次按键后，BUSY 灯会短暂熄灭
3. 在 BUSY 熄灭的这段时间里，无论单次还是连续按键，都更容易得到正确扫描码
4. 停止输入一段时间后，BUSY 灯恢复闪烁
5. BUSY 恢复闪烁后，再次第一次按键又容易没有有效显示
```

这个现象解释了为什么早期看起来只有“连续多次按键”才能成功：

```text
前几次按键实际上是在把板载 USB HID -> PS/2 仿真链路推进到稳定工作状态
真正稳定送进 FPGA 的，是后面几次按键
```

### 为什么第一次曾显示 AA

统一联调初期还有一个重要现象：

```text
第一次按键时，扫描码会显示为 AA
```

这说明第一次输入其实不是“完全没有数据”，而是：

```text
第一次进入 FPGA 的字节并不是普通按键的 make code
而更像是板载仿真链路送出的服务字节/初始化字节
```

结合 PS/2 键盘协议背景，这里的 `0xAA` 很像键盘自检通过（BAT OK）字节。于是我们把问题从“键盘连接不好”进一步收敛成：

```text
1. 板载辅助控制器会把 USB 键盘转换成 PS/2 风格信号送给 FPGA
2. 这个过程中首次输入可能先出现状态字节/服务字节
3. 软件侧如果直接把首次字节当普通扫描码处理，就会显示 AA
```

### 排查过程

1. 先排除 CPU 和 PLIC 是否完全失效

   因为在部分情况下中断次数能够增加，说明：

   ```text
   io_ps2 -> interrupt_controller -> CPU trap -> handler -> mret
   这条链路并不是完全断的
   ```

   所以问题更可能在 `io_ps2` 收到什么字节、什么时候把字节稳定交给 CPU，而不是 CPU 完全进不了中断。
2. 区分“速度慢”与“事件丢失”

   一开始也怀疑是 `CLK_DIV_BITS=18` 导致 CPU 太慢，肉眼看起来像“没反应”。但进一步分析后发现：

   ```text
   速度慢最多导致延迟
   不应该导致单次有效按键永久丢失
   ```

   因此“单次按键无反应、连续按键反而成功”不能单纯归因为分频太大。
3. 修正程序与观察链路

   为了排除显示逻辑本身的问题，我们先修正了 PS/2 demo 程序以及数码管扫描设置，确保：

   ```text
   LED[15:8] 显示中断次数
   LED[7:0]  显示最近扫描码
   SEG       显示最近扫描码
   ```

   同时把数码管扫描频率调回适合板上观察的范围，避免出现 LED 已更新、但数码管肉眼看似固定不变的假象。
4. 过滤服务字节

   在 `io_ps2.v` 中加入对 `0xAA` 和 `0xFA` 的过滤，不再把这类服务字节当作普通扫描码上报。

   这一步解决了最早“第一次按键显示 AA”的现象。
5. 修正事件保持机制

   继续分析后发现，原来的 `io_ps2` 更像是单字节暂存器：

   ```text
   当前字节还没被 CPU 读走时，后续事件容易覆盖或丢失
   ```

   这对“短按一次然后很快松开”“按下与释放靠得很近”“空闲后第一次输入”这几类场景尤其不利。

   因此又进一步把 `io_ps2` 改成了更稳的“事件邮箱”结构：

   ```text
   1. 当前扫描码保存在 rdata_byte
   2. 若 CPU 尚未读走，而新的扫描码到来，则先放入 shadow 槽
   3. CPU 读走当前字节后，shadow 中的下一个字节立即顶上来
   ```

### 根本原因

最终可以把这个问题分成两个层次：

```text
第一层：板载 USB HID -> PS/2 仿真链路在首次输入时可能先给出服务字节或状态变化
第二层：我们原来的 io_ps2 保持机制过弱，导致短按/首次输入时更容易丢失有效扫描码
```

也就是说，最开始把它完全理解成“键盘连接问题”并不准确；真正的问题是：

```text
外部输入链路本身存在状态字节和稳定期
而 RTL 对短事件和相邻事件的保持又不够强
两者叠加后，才表现成“要连续按很多次才成功”
```

### 解决方法

```text
1. 在 io_ps2 中过滤 AA、FA 等服务字节
2. 保留对 F0、E0 前缀的协议级处理
3. 增加 shadow 缓冲槽，避免单字节事件在 CPU 读取前丢失
4. 调整数码管扫描显示设置，确保板上现象与软件状态一致
5. 重新综合、实现、烧录，结合 BUSY 灯状态做板级验证



重新更换PS2键盘后完全解决。
```

### 验证结论

修正后，仿真已经能够稳定通过：

```text
PASS: ps2 irq demo passed led=021b seg=1b mem_code=0000001b mem_count=00000002 claim=00000002
```

板级调试中也明确得出了以下结论：

```text
1. 第一次输入并非完全无数据，而可能先出现 AA 这类服务字节
2. 连续输入之所以更容易成功，是因为板载仿真链路进入了稳定工作窗口
3. CPU 中断链路本身是通的，问题重点在 io_ps2 与板载输入链路的配合
```

### 经验总结

这次 PS/2 问题和之前的 MMIO/流水线问题不同，它提醒我们：

```text
1. 板上“外设有反应”不等于 FPGA 软件层已经拿到正确字节
2. 单测阶段的“偶发成功”不能简单视为键盘接触不良
3. 遇到首次输入异常、连续输入正常这类现象时，应优先考虑：
   - 服务字节是否被误当作有效数据
   - 输入事件是否在 CPU 读取前被覆盖
   - 板载桥接/仿真链路是否存在稳定窗口
```

这也是本轮调试里一个比较典型的例子：最开始的问题表面像“键盘不稳定”，但真正深入定位后，实际上是“板载 HID/PS2 仿真链路行为 + RTL 事件保持策略”共同造成的结果。

## PS/2 键盘中断 ISR 调试（2026-07-12）

为流水线 SoC 编写了完整的 PS/2 键盘中断服务程序，实现数字 0-9 在 7 段数码管上滚动显示，退格右移、回车清空。在调试过程中陆续发现并修复了以下问题：

### Bug 1：hex 文件 $readmemh 路径解析失败

**现象**：烧录后 LED 不亮，按键完全无反应。仿真显示 IMEM 所有单元为 0。

**原因**：soc.v 中 IMEM_INIT_FILE 使用相对路径 `sim/ps2_keyboard_isr.hex`，但 Vivado 综合/仿真的工作目录不是项目根，$readmemh 找不到文件，IMEM 未初始化。

**修复**：soc.v 改为绝对路径。

### Bug 2：汇编器 @ 地址格式错误

**现象**：LED 亮，但按键无反应。ISR 被加载到 IMEM 字索引 0x100（字节 0x400），但 mtvec=0x100 跳转到字索引 0x40（字节 0x100），CPU 中断时取到未初始化区域。

**原因**：$readmemh 将 @ADDR 解析为字索引，汇编器输出的是字节地址。.org 0x100 生成了 @100（字节 0x400），而非需要的 @40（字节 0x100）。

**修复**：汇编器输出改为 `@{a//4:X}`（字节地址除以 4 = 字索引）。

### Bug 3：io_seg7 显示扫描只驱动 2 位数码管

**现象**：LED 亮，键盘中断能正确触发，但数码管无任何变化。

**原因**：io_seg7.v 的显示扫描 case(scan_index) 只处理了 0 和 1 两个位置（用 hex_value 驱动），raw_digits[0..7] 虽然能被 CPU 写入，但扫描逻辑从未读取它们来驱动显示。

**修复**：扫描 case 改为 8 个分支，每个读取 raw_digits[i][3:0] 驱动对应数码管位。

### Bug 4：dmem_bus_decoder SEG 地址范围不够

**现象**：AN7（最左位数码管）始终显示 F，其余位数码管正常。

**原因**：dmem_bus_decoder.v 中 hit_seg 范围为 0x80000010 ~ 0x8000002F，但 DIGIT7 地址是 0x80000030，刚好超出，落入 hit_btn 范围，导致写入被路由到只读的按钮模块，raw_digits[7] 保持复位值 0xFF。

**修复**：hit_seg 扩展到 0x33，hit_btn 后移至 0x34 ~ 0x37。

### 经验总结

本次调试的几个 bug 都不是汇编逻辑层面，而是工具链和硬件基础设施问题：

```text
1. 相对路径在 Vivado 中不可靠 → 用绝对路径
2. $readmemh 的 @ 地址是字索引而非字节地址
3. 修改外设 RTL 时要检查显示扫描逻辑是否实际读取了写入的寄存器”共同造成的结果。
4. 总线解码器的地址范围必须覆盖外设内部所有寄存器，否则数据路由到错误模块
```
