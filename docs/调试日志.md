# 调试日志

## Nexys 4 DDR 无法被 Vivado 识别问题

### 问题现象

在 Vivado 2018.3 的 Hardware Manager 中连接 Nexys 4 DDR 时，最开始只能看到 `localhost` 硬件服务器，无法打开具体 FPGA 设备。界面中出现过以下现象：

```text
No hardware target is open
localhost (0)
No hardware targets exist on the server [localhost:3121]
```

在安装 Digilent 驱动并调整跳线后，Vivado 能看到 Digilent 下载目标，但仍然提示 JTAG 链上没有 FPGA 设备：

```text
localhost (1)
xilinx_tcf/Digilent/...
No devices detected on target
```

这说明问题分成两个阶段：

```text
第一阶段：电脑没有正确识别 Digilent USB-JTAG 下载器
第二阶段：电脑识别到了下载器，但 JTAG 链没有扫到 FPGA 芯片
```

### 排查过程

1. 检查 Vivado Hardware Manager

   先确认 `hw_server` 是否启动，以及 `localhost` 是否连接。发现 `localhost` 可以连接，但最初目标数为 0，说明 Vivado 本身启动正常，问题更可能在 USB 设备、驱动或板卡连接链路。

2. 检查 Nexys 4 DDR 跳线

   根据 Digilent 入门文档和老师提示，重点检查 JP1 和 JP2：

   ```text
   JP1 / MODE：需要设置到 JTAG，才能通过 Vivado Hardware Manager 下载 bitstream
   JP2：与 SD/USB 启动相关，本次 JTAG 下载不是主要矛盾
   ```

   一开始 JP1 处于 QSPI 模式，此时开发板会运行 Flash 中的出厂程序，七段数码管出现类似“贪吃蛇”的移动光效。切换到 JTAG 后，出厂程序不再自动运行，只剩电源指示灯亮，这是正常现象，并不代表板子损坏。

3. 安装/重装 Digilent 相关驱动

   根据 Vivado 和 Digilent 的说明，安装 Digilent Adept Runtime，并检查 Windows 设备管理器中是否存在异常 USB 设备。过程中曾出现：

   ```text
   未知 USB 设备（设备描述符请求失败）
   ```

   这类提示说明 Windows 已经检测到有 USB 设备插入，但无法正确读取设备描述符，问题可能来自驱动、USB 口、USB 线或板卡 USB 接口。

4. 对比不同 USB 口和连接方式

   更换过电脑 USB 口，并反复执行 Vivado 中的 `Open Target`、`Auto Connect`、`Refresh Server` 等操作。虽然连接状态有变化，但仍然不能稳定识别 FPGA 设备，因此排除了单纯 Vivado 操作步骤错误的可能。

5. 对比其他小组的成功现象

   其他小组能够正常下载流水灯 bitstream，说明同一型号开发板和 Vivado 流程本身是可行的。于是把重点转向本地连接链路，包括 USB 线材、供电和数据传输能力。

### 根本原因

最终发现问题出在 Nexys 开发板使用的原装 Micro-USB 线。该线在当前环境下可以给开发板供电，但无法稳定进行 USB 数据传输，导致 Vivado 无法通过 Digilent USB-JTAG 链路识别 FPGA。

换用一根确认支持数据传输的 Micro-USB 线后，Vivado 能够正常识别开发板，后续可以进行 bitstream 下载验证。

### 解决方法

```text
1. 使用支持数据传输的 Micro-USB 线连接 Nexys 4 DDR 的 PROG/UART 接口
2. 确认电源开关打开
3. 确认 JP1 / MODE 设置为 JTAG
4. 打开 Vivado Hardware Manager
5. 选择 Open Target -> Auto Connect
6. 识别到 FPGA 后，选择 Program Device 下载 bitstream
```

### 验证结果

更换 USB 线后，可以继续进行板卡验证。先使用独立的流水灯 bitstream 验证下载链路：

```text
build/bitstreams/led_chaser_top.bit
```

流水灯能够运行，说明：

```text
USB 数据线正常
Digilent 驱动正常
Vivado Hardware Manager 能够识别并配置 FPGA
JTAG 下载链路正常
开发板基本硬件状态正常
```

随后可下载第一层 CPU bitstream：

```text
build/bitstreams/fpga_top.bit
```

通过拨码开关观察 CPU 测试程序结果：

```text
SW[1:0] = 01：观察 x5，期望为 0001
SW[1:0] = 10：观察 data_mem[0]，期望为 000c
```

### 经验总结

本次问题的关键不是 Verilog 代码或 bitstream 生成错误，而是硬件下载链路中的 USB 数据传输失败。Micro-USB 线即使能供电，也不一定支持或稳定支持数据通信。遇到 Vivado 无法识别 FPGA 时，应按以下顺序排查：

```text
1. 板卡供电是否正常
2. JP1 是否处于 JTAG 模式
3. Windows 设备管理器是否识别到 Digilent/USB Serial 设备
4. Vivado Hardware Manager 是否能看到 hw_server 和 target
5. USB 线是否支持数据传输
6. Digilent/Vivado cable driver 是否正确安装
```

其中 USB 线材问题最容易被忽略，因为“能供电”会让人误以为线一定没有问题。

## CPU + MMIO 联调显示异常问题

### 问题现象

在完成 `CPU + imem + dmem_bus_decoder + LED/SW/SEG` 联调并烧录上板后，最开始出现过以下现象：

```text
1. LED 和数码管最初都不响应 SW
2. 后续 LED 可以变化，但数码管始终显示 0
3. 再继续修改后，板上又出现 LED 熄灭、数码管全 0 的情况
```

表面上看像是 I/O 模块或顶层连接问题，但由于 LED、SEG、SW 三个外设并不是同时失效，所以更怀疑是 CPU 的执行路径、取指握手或测试程序本身有问题。

### 排查过程

1. 先检查顶层 MMIO 通路

   确认 `fpga_top` 中 `dmem_bus_decoder`、`io_switches`、`io_leds`、`io_seg7` 的片选和连接关系正确，排除明显接线错误。

2. 从上板观察切换到仿真定位

   仅靠 LED 和数码管现象很难判断到底是哪一级出错，因此转到 Vivado 仿真。这里我借助 AI 工具一起梳理了需要重点观察的信号，包括：

   ```text
   pc
   if_id_valid / id_ex_valid / mem_wb_valid
   dmem_addr / dmem_wdata
   x2 寄存器值
   ```

   同时补了 `tb_pipeline_mmio_mirror.v`，专门验证 `lw SW -> sw LED -> sw SEG` 这条最小 MMIO 路径。

3. 发现取指等待不应冻结整个后端流水线

   仿真中发现，当前 CPU 在 `imem_wait` 时把后端流水线也一起停住，导致已经进入 EX/MEM/WB 的老指令推进异常。于是把停顿条件拆开：

   ```text
   pipeline_stall = dmem_wait || load_use_stall
   perf_stall     = imem_wait || pipeline_stall
   ```

   这样取指等待只影响前端，不再错误阻塞后端提交。

4. 发现跳转后旧取指响应被错误写回 IF/ID

   仿真继续显示 `jal` 回跳后 CPU 仍可能取到旧 PC 对应的返回指令。为此在 CPU 中增加了 `discard_imem_resp`，在控制流重定向时丢弃飞行中的旧取指响应，避免错误路径指令重新进入流水线。

5. 发现 demo 程序本身也有问题

   为了测试 load 写回，我之前在 demo ROM 中插入了两个 `nop`，但后面的 `jal` 偏移没有同步修改，导致程序回跳位置错误，没有真正重新执行 `lw SW`。这一点也是在 AI 辅助下，通过仿真时序一起定位出来的。

6. 修正板级 demo IMEM 时序

   `fpga_demo_imem.v` 原本把 `imem_ack` 和 `imem_data` 在同一拍寄存输出，和正式 `imem.v` 的响应行为不一致，容易放大取指时序问题。后续把它改成和正式 `imem` 一致的响应方式，再继续验证。

### 根本原因

这次问题不是单一模块故障，而是三个因素叠加：

```text
1. CPU 取指等待逻辑过于保守，错误冻结后端流水线
2. 控制流重定向时，没有丢弃旧的取指返回
3. 板级 demo 程序插入 nop 后，jal 回跳偏移没有同步修正
```

因此板上才会出现“有时 LED 可变、有时数码管全 0、行为前后不一致”的现象。

### 解决方法

```text
1. 修改 pipeline_cpu_top.v，拆分 imem_wait 与真正的 pipeline_stall
2. 在 CPU 中加入 discard_imem_resp，丢弃跳转后的旧取指响应
3. 修改 fpga_demo_imem.v 的返回时序，使其与正式 imem 行为一致
4. 修正 demo 程序中 jal 的回跳偏移
5. 通过 mmio 定向 testbench 反复验证后，再重新综合烧录
```

### 验证结果

仿真最终通过：

```text
SW = 3 -> LED = 0003, SEG = 03
SW = 2 -> LED = 0002, SEG = 02
PASS: MMIO mirror pipeline path passed
```

重新烧录后，板上 LED 和数码管也已经能够正常随开关变化，说明：

```text
CPU 的 lw/sw 基本执行路径正确
MMIO 地址译码正确
LED/SEG 外设写通路正确
当前板级 demo 可以作为后续中断联调的基础版本
```

### 经验总结

这次调试说明，SoC 联调里“显示异常”不一定是显示模块自己的问题，很多时候真正的问题在 CPU 流水线控制或测试程序本身。后续再遇到类似问题，优先按下面顺序排查：

```text
1. 先确认测试程序是否正确
2. 再确认 MMIO 地址译码和顶层连接
3. 最后用仿真看 CPU 内部推进、写回和跳转逻辑
```

另外，这次使用 AI 工具的主要帮助不是直接替我写代码，而是协助我把问题拆成“顶层连接、仿真信号、流水线停顿、控制流重定向、测试程序”几个层次逐步定位，明显提高了排错效率。
