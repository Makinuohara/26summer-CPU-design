# 阶段二分工说明

## 1. 阶段二目标

阶段二目标是在当前基础单周期 CPU 工程上，推进到“完整计算机系统设计”和“流水线 CPU 子系统”的代码与验证交付。验收时需要能够说明系统结构、地址映射、内存层次、板载 I/O、CPU 子系统实现和性能评估。

当前工作分支：

```text
stage2/work-division
```

## 2. 总体分工

| 成员 | 方向 | 核心目标 | 主要交付物 |
| --- | --- | --- | --- |
| A | 系统结构设计 | 设计计算机系统的宏观结构，定义模块接口契约 | 系统结构图、地址映射表、`soc_top`、`fpga_top`、集成测试版本 |
| B | 内存子系统 | 完成存储层次规划，规范 `imem/dmem`，尝试小型 Cache | `ram`、`imem`、`dmem`、`cache`、hit/miss 统计、访存瓶颈分析 |
| C | 板载 I/O 接口集成 | 让系统使用 Nexys 4 的 LED、拨码开关、数码管等 I/O 设备 | I/O 地址说明、LED/SW/数码管模块、测试程序、演示脚本 |
| D | CPU 子系统 | 实现 RISC-V32I 子集，推进流水线机制与性能评估 | `pipeline_cpu_top`、流水线结构图、CPI/stall/flush 统计、频率对比 |

## 3. 推荐代码结构

阶段二建议在现有 `src/` 基础上增加更清晰的系统分层：

```text
src/
  cpu/
    single_cycle/        当前基础 CPU 可逐步迁入或保留对照
    pipeline/            D 负责，流水线 CPU 子系统
  mem/
    imem.v               B 负责，指令存储器
    dmem.v               B 负责，数据存储器
    ram.v                B 负责，通用 RAM
    cache.v              B 负责，可选小型 Cache
  io/
    gpio_led.v           C 负责，LED 输出
    gpio_sw.v            C 负责，拨码开关输入
    seg7_display.v       C 负责，数码管显示
  soc/
    address_map.vh       A 负责，地址映射定义
    soc_top.v            A 负责，系统级顶层
  fpga_top.v             A/C 联合维护，上板顶层
```

当前工程还没有完成上述目录拆分。为了降低风险，建议先保留现有 `src/*.v` 不动，在阶段二功能稳定后再做迁移。

## 4. A：系统结构设计

### 4.1 工作范围

A 负责定义整个系统的顶层结构和模块连接方式，重点是把 CPU、内存和 I/O 组织成一个可集成、可验证的系统。

### 4.2 关键需求

| 编号 | 需求 | 说明 |
| --- | --- | --- |
| A-01 | 设计系统结构图 | 说明 CPU、imem、dmem、I/O、总线或地址译码之间的关系 |
| A-02 | 制定地址映射 | 明确 RAM、LED、SW、数码管、统计寄存器等地址区间 |
| A-03 | 定义模块接口 | 统一时钟、复位、地址、写使能、读数据、写数据等信号 |
| A-04 | 实现 `soc_top` | 负责连接 CPU、内存子系统和 I/O 模块 |
| A-05 | 维护 `fpga_top` | 将 `soc_top` 接入 Nexys 4 DDR 板级端口 |
| A-06 | 集成测试版本 | 汇总 B/C/D 模块，保证顶层可综合、可仿真 |

### 4.3 建议地址映射

| 地址范围 | 模块 | 说明 |
| --- | --- | --- |
| `0x0000_0000` - `0x0000_0fff` | 指令/数据 RAM | 小型测试程序和数据区 |
| `0x1000_0000` | LED | 写低 16 位控制 LED |
| `0x1000_0004` | SW | 读低位拨码开关状态 |
| `0x1000_0008` | 七段数码管 | 写显示数据 |
| `0x1000_0010` | 性能计数器 | 可读 cycle、instret、stall、flush 等统计 |

## 5. B：内存子系统

### 5.1 工作范围

B 负责规划和实现存储层次，先保证 `imem/dmem` 稳定可用，再尝试小型 Cache 或访存统计。

### 5.2 关键需求

| 编号 | 需求 | 说明 |
| --- | --- | --- |
| B-01 | 重构指令存储器 | 支持从 hex 文件初始化，便于替换测试程序 |
| B-02 | 重构数据存储器 | 支持统一地址访问和 word 读写 |
| B-03 | 规划存储层次 | 明确 imem、dmem、RAM、可选 Cache 的关系 |
| B-04 | 实现访存统计 | 统计访存次数、命中次数、未命中次数 |
| B-05 | 尝试小型 Cache | 可实现直接映射 Cache，作为拓展交付 |
| B-06 | 分析访存瓶颈 | 说明访存延迟、Cache 命中率对系统性能的影响 |

### 5.3 最小交付标准

1. `imem` 能通过 hex 初始化运行测试程序。
2. `dmem` 能支持 CPU 的 `lw/sw` 访问。
3. 内存模块接口能被 `soc_top` 调用。
4. 至少提供一个访存测试程序和仿真结果。

## 6. C：板载 I/O 接口集成

### 6.1 工作范围

C 负责把 Nexys 4 DDR 的 LED、拨码开关、数码管等板载设备接入系统，使 CPU 程序能够通过 I/O 地址读写外设。

### 6.2 关键需求

| 编号 | 需求 | 说明 |
| --- | --- | --- |
| C-01 | 定义 I/O 地址说明 | 与 A 的地址映射保持一致 |
| C-02 | 实现 LED 输出模块 | CPU 写指定地址后更新 LED |
| C-03 | 实现 SW 输入模块 | CPU 读指定地址得到拨码开关状态 |
| C-04 | 实现数码管显示模块 | CPU 写指定地址后更新显示值 |
| C-05 | 编写 I/O 测试程序 | 程序能读开关、写 LED 或数码管 |
| C-06 | 准备演示脚本 | 明确上板拨码、复位、观察结果步骤 |

### 6.3 最小交付标准

1. CPU 能通过 `sw` 指令写 LED 或数码管。
2. CPU 能通过 `lw` 指令读取拨码开关。
3. 上板后能观察到程序控制 I/O 的效果。

## 7. D：CPU 子系统

### 7.1 工作范围

D 负责 CPU 内核扩展和流水线化。当前基础 CPU 已实现 RV32I 子集的单周期版本，阶段二应补充常用指令，并推进流水线 CPU。

### 7.2 关键需求

| 编号 | 需求 | 说明 |
| --- | --- | --- |
| D-01 | 扩展 RV32I 指令 | 优先实现移位、比较、`lui`、`auipc`、`jalr` 等 |
| D-02 | 完善分支指令 | 支持 `blt`、`bge`、`bltu`、`bgeu` |
| D-03 | 设计流水线结构 | 明确 IF、ID、EX、MEM、WB 五级划分 |
| D-04 | 实现流水线寄存器 | 添加 IF/ID、ID/EX、EX/MEM、MEM/WB |
| D-05 | 处理流水线冒险 | 实现 stall、flush、forwarding |
| D-06 | 增加性能统计 | 统计 cycle、instruction、stall、flush |
| D-07 | 输出性能对比 | 对比单周期和流水线的 CPI、频率、吞吐量 |

### 7.3 最小交付标准

1. 阶段二 CPU 能运行基础测试程序并保持正确结果。
2. 流水线版本能处理基础数据相关和分支跳转。
3. 提供 CPI、stall、flush 的仿真统计结果。
4. 提供 Vivado 频率或资源占用对比。

## 8. 集成顺序

推荐按以下顺序推进，减少多人合并冲突：

1. A 先冻结 `soc_top` 接口和地址映射。
2. B 将 `imem/dmem` 接入 `soc_top`，保证 CPU 能运行原测试程序。
3. C 接入 LED/SW/数码管，完成 memory-mapped I/O。
4. D 在保持接口兼容的前提下替换或新增 `pipeline_cpu_top`。
5. A 汇总集成测试，确认仿真和 Vivado 构建脚本可用。
6. 全组统一准备验收程序、演示流程和性能表格。

## 9. 验收代码要求

阶段二完整代码至少应满足：

1. 有一个明确的系统级顶层，例如 `soc_top` 或新版 `fpga_top`。
2. CPU、内存、I/O 模块之间接口清晰，不依赖临时硬编码调试线完成主要功能。
3. 测试程序能通过指令存储器初始化或构建脚本加载。
4. 仿真能验证 CPU 执行、访存和 I/O 地址访问。
5. Vivado 能完成综合、实现并生成 bitstream。
6. 报告能对应说明 A/B/C/D 各自交付物。
